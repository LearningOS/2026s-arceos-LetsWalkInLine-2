# sys_mmap 练习笔记

## 1) 练习目标

本次练习的目标是：

- 在 `arceos/exercises/sys_map/src/syscall.rs` 中实现 `sys_mmap`；
- 让用户程序能够通过 `mmap(...)` 将一段文件内容映射到自己的虚拟地址空间；
- 通过测试脚本：`./scripts/test-sys_map.sh`。

这个练习的核心不是“把文件读出来”这么简单，而是要理解：

- 什么是虚拟内存映射；
- `mmap` 的参数分别代表什么；
- 内核如何为用户进程挑选虚拟地址；
- 内核如何把“文件内容”和“虚拟地址空间”联系起来。


## 2) 我的思考过程

### 2.1 最开始的直觉

一开始我对这题的理解是：

- `sys_mmap` 应该主要做参数解析；
- 真正的映射动作应该复用 `axmm` 里的现成能力；
- 既然 `axmm` 里有 `map_alloc`，那应该直接拿来用。

这个方向总体是对的，但后面我逐渐意识到：

- 真正应该调用的是 `AddrSpace` 对外提供的 `map_alloc(...)`；
- 而不是去碰 `backend/mod.rs` 里面的底层实现细节。

原因是：

- `backend` 是内存映射后端实现，属于 `axmm` 内部抽象；
- `AddrSpace` 才是用户地址空间管理的正式入口；
- 练习代码本身持有的也是当前任务的 `aspace`，所以顺着这个对象调用最自然。


### 2.2 第二个误区：以为 mmap 只是“分配内存”

刚开始很容易把 `mmap` 想成：

- “给用户程序分一块内存”。

但实际上它比 `malloc` 更底层，也更靠近操作系统：

- `malloc` 更像是语言运行时/库层面的堆分配；
- `mmap` 是内核在“进程虚拟地址空间”里建立一段映射。

也就是说，`mmap` 并不是简单返回一个可用指针，而是做了两件事：

1. 在虚拟地址空间里预留一段区间；
2. 为这段区间建立页表映射，并赋予访问权限。

如果是文件映射，还要进一步考虑：

3. 这段虚拟内存中应该出现什么初始内容。


### 2.3 第三个关键认识：本题并不要求完整 Linux mmap

我后来去看了测试程序 `arceos/payload/mapfile_c/mapfile.c`，发现它只用了这一种场景：

```c
addr = mmap(NULL, 32, PROT_READ, MAP_PRIVATE, fd, 0);
```

这说明测试需求是一个“最小可用子集”：

- 地址由内核自己选（`addr = NULL`）；
- 映射长度很小（32 字节）；
- 只读映射（`PROT_READ`）；
- 私有文件映射（`MAP_PRIVATE`）；
- 从文件开头开始（`offset = 0`）。

于是我明确了实现策略：

- 先做一个“足够正确、足够清晰、能通过练习”的版本；
- 不追求一次性覆盖 Linux `mmap` 的所有语义。


## 3) 先补背景：什么是 sys_mmap

对于一个完全没接触过 `mmap` 的人，可以把它理解成：

> 向内核申请：请在我的进程虚拟地址空间里，划出一段地址，并让这段地址表现得像某种内存。

这里的“像某种内存”，可能有两种来源：

- **匿名内存**：背后不是文件，只是普通内存页；
- **文件映射**：背后对应一个文件，访问这段内存时，看到的是文件内容。

### 3.1 一个非常形象的类比

可以把进程虚拟地址空间想成一本巨大的“空白地址簿”。

- `malloc` 更像是在地址簿上找一块给程序内部堆用；
- `mmap` 则像是向内核申请：
  - 在地址簿的某个范围贴上一张标签；
  - 标签写着“这里可以读/写/执行”；
  - 这块区域的数据可能来自文件，也可能来自新分配的物理页。


## 4) mmap 六个参数分别是什么意思

`mmap` 的经典原型是：

```c
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
```

### 4.1 `addr`

- 表示“希望映射到哪里”；
- 如果传 `NULL`，表示“由内核自己挑一块合适的虚拟地址”；
- 如果配合 `MAP_FIXED`，则表示“必须用我指定的这个地址”。

本题测试里传的是 `NULL`，所以内核自己找空闲区间就可以。


### 4.2 `length`

- 表示要映射多少字节；
- 但底层页表是按页工作的，所以内核通常会向上对齐到页大小（例如 4 KiB）。

比如：

- 用户说要 32 字节；
- 内核实际可能会映射 4096 字节；
- 只是返回给用户的起始地址仍然可以当成 32 字节范围来使用。


### 4.3 `prot`

- 表示这段映射区的访问权限；
- 常见取值：
  - `PROT_READ`：可读
  - `PROT_WRITE`：可写
  - `PROT_EXEC`：可执行

这些权限最终会被转换成页表项中的权限位。

在本练习里，代码里已经定义了：

- `MmapProt`
- `impl From<MmapProt> for MappingFlags`

所以只要把用户传入的 `prot` 解析后转成 `MappingFlags` 即可。


### 4.4 `flags`

- 表示映射行为；
- 比如是共享还是私有，是不是匿名映射，是不是固定地址映射。

本题里最重要的几个 flag：

- `MAP_PRIVATE`：私有映射；
- `MAP_SHARED`：共享映射；
- `MAP_ANONYMOUS`：匿名映射，不依赖文件；
- `MAP_FIXED`：强制映射到指定地址。

当前测试只用到了 `MAP_PRIVATE`。

不过这里最值得重点讲清的是：

- `MAP_SHARED`
- `MAP_PRIVATE`

因为这两个标志决定了“这段映射和底层文件之间到底是什么关系”。

#### `MAP_SHARED` 到底是什么意思

`MAP_SHARED` 可以理解为：

> 这段映射内存和底层文件是“共享视图”。

也就是说，理想语义下：

- 如果进程修改了映射区内容；
- 那么这些修改应当能够反映回文件；
- 同时，其他映射了同一文件同一区域的进程，也可能观察到这些变化。

你可以把它想成：

- 文件像是一份“公共文档”；
- 多个进程通过 `MAP_SHARED` 打开的是同一份内容的共享窗口；
- 某个人在窗口里改内容，本质上是在改公共文档本体。

所以 `MAP_SHARED` 的关键字是：

- **共享底层对象**；
- **修改可能回写文件**；
- **多个映射者之间可能互相可见**。


#### `MAP_PRIVATE` 到底是什么意思

`MAP_PRIVATE` 可以理解为：

> 这段映射最初看起来来自文件，但后续修改是“我自己的”，不应该改动底层文件。

也就是说，理想语义下：

- 映射建立时，初始内容来自文件；
- 读的时候像是在读文件内容；
- 但如果进程对这段映射进行了写入，这些修改不应该回写到底层文件；
- 其他进程也不应该看到这些私有修改。

这通常通过 **写时复制（Copy-On-Write, COW）** 来实现：

- 一开始多个映射可以共享同一份物理页；
- 只有当某个进程真的去写时；
- 内核才为它复制出一份新的私有页；
- 从那一刻开始，它改的是自己的副本，而不是原文件。

所以 `MAP_PRIVATE` 的关键字是：

- **初始内容来自文件**；
- **后续修改是私有的**；
- **通常依赖 COW 实现**；
- **不会直接回写底层文件**。


#### `MAP_SHARED` 和 `MAP_PRIVATE` 的联系

这两个标志虽然行为不同，但它们有一个共同点：

- 都是在描述“文件映射和底层对象之间的关系”。

更具体地说：

- 两者都可以让映射区的初始内容来自文件；
- 两者都属于“文件映射”的范畴；
- 区别不在于“是不是来自文件”，而在于“后续修改如何传播”。

所以它们的共同起点是：

- **都可以把文件内容呈现在进程地址空间中**。

而它们的根本区别是：

- **写入之后，修改是共享出去，还是只留给自己。**


#### 两者的直观对比

可以用一个简单表述来记忆：

- `MAP_SHARED`：我改了，大家和文件都可能看到。
- `MAP_PRIVATE`：我改了，只有我自己看到，文件不该被改。

再换一个更生活化的比喻：

- `MAP_SHARED` 像在一块公共白板上写字；
- `MAP_PRIVATE` 像先拿到白板内容的复印件，再在自己的副本上做标注。


#### 为什么系统调用里要求二选一

这也解释了为什么实现里要检查：

- `MAP_SHARED` 和 `MAP_PRIVATE` 必须二选一。

因为它们回答的是同一个问题：

- “这段映射的修改传播策略是什么？”

而这个问题不可能同时有两个答案：

- 不可能既要求“修改对大家可见并回写文件”；
- 又要求“修改只对自己可见且不回写文件”。

所以从语义上，它们本来就是互斥的。


#### 为什么本练习虽然只用了 `MAP_PRIVATE`，但没有实现真正的 COW

这一点非常重要。

本练习中的实现是一个“教学版最小实现”：

- 我们确实解析了 `MAP_PRIVATE`；
- 也确实把文件内容放进了映射区；
- 但并没有进一步实现完整的写时复制机制。

原因是完整的 `MAP_PRIVATE` 往往需要：

- 页级别共享；
- 缺页异常处理；
- 写保护页；
- 在写入时复制新页；
- 维护文件页和私有页的关系。

这些都明显超出了当前练习的范围。

所以当前实现中的 `MAP_PRIVATE` 更准确地说是：

- **把文件内容拷贝到一段新映射里供当前进程使用**。

它足以满足本练习“读取文件映射内容”的目标，但还不是完整操作系统语义下的 `MAP_PRIVATE`。


#### `MAP_ANONYMOUS` 和前两个标志是什么关系

`MAP_ANONYMOUS` 表示：

- 这段映射不来自文件；
- 它只是普通匿名内存；
- 一般会忽略 `fd`，通常也要求 `offset` 没有实际意义。

这时：

- `MAP_SHARED` / `MAP_PRIVATE` 仍然可以表示“修改传播方式”；
- 但背后就不再是一个普通文件，而是一段匿名页。

在很多入门场景里，可以先这样理解：

- `MAP_ANONYMOUS` 决定“内容来源是不是文件”；
- `MAP_SHARED` / `MAP_PRIVATE` 决定“修改是共享还是私有”。


#### `MAP_FIXED` 的定位和前面几个标志不同

`MAP_FIXED` 不描述“内容共享语义”，它描述的是：

- **地址选择策略**。

也就是它回答的问题是：

- “这段映射一定要放在我指定的地址吗？”

所以：

- `MAP_SHARED` / `MAP_PRIVATE` 关心的是“映射行为”；
- `MAP_ANONYMOUS` 关心的是“内容来源”；
- `MAP_FIXED` 关心的是“地址是否固定”。

这几个 flag 是在不同维度上描述同一个映射请求的。


### 4.5 `fd`

- 文件描述符；
- 如果是文件映射，就表示“从哪个文件映射”；
- 如果是匿名映射，通常会忽略它。

本题测试中，`fd` 来自前面 `open(fname, O_RDONLY)` 返回的文件描述符。


### 4.6 `offset`

- 表示从文件的哪个偏移开始映射；
- 不是从头开始映射时，就要用这个参数；
- 在真正的 `mmap` 语义里，它通常要求页对齐。

本题测试传的是 `0`，也就是从文件开头开始映射。


## 5) 这题到底要做什么

从效果上看，测试程序的需求可以概括为：

1. 打开一个文件；
2. 这个文件里有字符串 `"hello, arceos!"`；
3. 调用 `mmap(NULL, 32, PROT_READ, MAP_PRIVATE, fd, 0)`；
4. `mmap` 返回一个用户虚拟地址；
5. 用户程序从这个地址开始读字符串，应该能读到文件内容。

所以内核实现上至少要完成：

1. 找一段用户地址空间里的空闲虚拟地址；
2. 建立页表映射；
3. 把文件内容拷进这段映射对应的物理页；
4. 把虚拟地址返回给用户。


## 6) 先读项目结构后的关键发现

本次实现涉及的关键对象有：

- `current().task_ext().aspace`：当前用户任务的地址空间；
- `AddrSpace::find_free_area(...)`：找空闲虚拟地址区间；
- `AddrSpace::map_alloc(...)`：创建“分配型映射”；
- `AddrSpace::write(...)`：把数据写进用户地址空间；
- `api::get_file_like(fd)`：通过文件描述符拿到文件对象。

这里最重要的理解是：

- `sys_mmap` 并不直接操作裸页表；
- 而是借助 `AddrSpace` 这层抽象来完成地址空间管理。

这种设计非常合理，因为：

- 地址空间的范围检查、区域管理、页表操作都封装在 `axmm` 中；
- 系统调用层只需要表达“我要映射什么”，而不必重复发明内存管理细节。


## 7) 最终实现的总体思路

最终我把 `sys_mmap` 的逻辑拆成 9 个步骤：

1. 校验基础参数；
2. 解析 `prot` 和 `flags`；
3. 检查 `MAP_SHARED` / `MAP_PRIVATE` 的合法性；
4. 检查 `offset` 合法性；
5. 将长度向上按页对齐；
6. 选择映射起始虚拟地址；
7. 创建映射并分配物理页；
8. 如果是文件映射，则把文件内容写入映射区；
9. 返回映射起始地址。

下面详细讲解每一步。


## 8) 最终实现详解

实现文件：`arceos/exercises/sys_map/src/syscall.rs:138`

### 8.1 第 1 步：校验 `length`

首先检查：

```rust
if length == 0 {
    return Err(LinuxError::EINVAL);
}
```

原因很简单：

- 映射长度不能为 0；
- 长度为 0 没有实际意义；
- 一般系统调用会返回 `EINVAL`（无效参数）。


### 8.2 第 2 步：解析 `prot` 和 `flags`

代码中使用：

```rust
let prot = MmapProt::from_bits(prot).ok_or(LinuxError::EINVAL)?;
let mmap_flags = MmapFlags::from_bits(flags).ok_or(LinuxError::EINVAL)?;
```

这里做的事是：

- 把用户传进来的整数位掩码，解释成内部的 bitflags 类型；
- 如果用户传了未定义的位，就返回 `EINVAL`。

这一步很关键，因为后面所有逻辑都依赖对参数语义的正确理解。


### 8.3 第 3 步：检查 `MAP_SHARED` / `MAP_PRIVATE`

这里我做了：

```rust
let is_shared = mmap_flags.contains(MmapFlags::MAP_SHARED);
let is_private = mmap_flags.contains(MmapFlags::MAP_PRIVATE);
if is_shared == is_private {
    return Err(LinuxError::EINVAL);
}
```

含义是：

- 这两个选项必须二选一；
- 两个都不选，不合法；
- 两个都选，也不合法。

为什么？

- `MAP_SHARED` 表示共享映射；
- `MAP_PRIVATE` 表示私有映射；
- 这两个行为本质上是互斥的。

本题测试只会传 `MAP_PRIVATE`，但这一步检查仍然值得写，因为它体现了系统调用的基本健壮性。


### 8.4 第 4 步：检查 `offset`

代码中：

```rust
if offset < 0 || (offset as usize) & (PAGE_SIZE_4K - 1) != 0 {
    return Err(LinuxError::EINVAL);
}
```

这一步做了两个约束：

- 偏移不能为负；
- 偏移必须按页对齐。

为什么要按页对齐？

- 因为页表映射的最小粒度通常是一页；
- 文件映射也是按页组织更合理；
- 所以很多系统都要求 `offset` 是页大小的倍数。

在本题测试里，`offset = 0`，自然是合法的。


### 8.5 第 5 步：把长度按页对齐

辅助函数：

```rust
fn align_up_4k(size: usize) -> Result<usize, LinuxError>
```

作用是：

- 把用户请求的长度向上补齐到 4 KiB；
- 同时用 `checked_add` 防止整数溢出。

比如：

- `length = 32`
- 对齐后 `aligned_len = 4096`

这样后续做页表映射时才是合法的。


### 8.6 第 6 步：选择映射起始地址

这是整个 `mmap` 中非常核心的一步。

代码里分成两种情况：

#### 情况 A：`MAP_FIXED`

```rust
let fixed_addr = VirtAddr::from(addr as usize);
if !fixed_addr.is_aligned_4k() {
    return Err(LinuxError::EINVAL);
}
```

如果用户要求固定地址映射：

- 就使用 `addr` 指定的虚拟地址；
- 但必须页对齐，否则非法。

#### 情况 B：非 `MAP_FIXED`

```rust
let hint = if addr.is_null() {
    aspace.base()
} else {
    VirtAddr::from(addr as usize).align_down_4k()
};
let limit = VirtAddrRange::from_start_size(aspace.base(), aspace.size());
aspace.find_free_area(hint, aligned_len, limit)
```

如果不要求固定地址：

- `addr = NULL` 时，就从用户地址空间起始位置开始找；
- `addr != NULL` 时，把它当作一个 hint（提示地址）；
- 再调用 `find_free_area(...)` 找一块足够大的空闲区间。

这一段很值得理解：

- 用户进程的地址空间不是一整块完全空白区域；
- 里面可能已经有代码段、数据段、栈等映射；
- 所以 `mmap` 不能乱选地址，必须找一段还没有被占用的区域。


### 8.7 第 7 步：建立映射

代码中：

```rust
let map_perm: MappingFlags = prot.into();
aspace.map_alloc(map_start, aligned_len, map_perm, true)
```

这里发生的事是：

1. 把 `prot` 转成页表权限位；
2. 在地址空间 `aspace` 中创建一段分配型映射；
3. `populate = true`，表示立即分配物理页。

为什么这里用 `true`？

- 因为本练习没有额外注册 page fault handler 来处理 lazy allocation；
- 如果这里不立即分配，后续用户一访问映射区就可能触发缺页异常而无人处理；
- 所以当前实现采用 eager allocation 是最直接可靠的。

可以把这一行理解成：

- “在用户虚拟地址 `map_start ~ map_start + aligned_len` 这段区域，建立读/写/执行权限对应的页表映射，并给它准备好物理页”。


### 8.8 第 8 步：把文件内容写进映射区

这是本练习最容易被忽视，但又最关键的一步。

因为：

- `map_alloc` 只负责“给你一段已经映射好的内存”；
- 它并不会自动知道这块内存应该装什么文件内容；
- 所以必须由 `sys_mmap` 自己把文件数据读出来，再写进映射区。

代码逻辑是：

```rust
if !mmap_flags.contains(MmapFlags::MAP_ANONYMOUS) {
    if fd < 0 {
        return Err(LinuxError::EBADF);
    }

    let file = api::get_file_like(fd)?;
    let mut data = vec![0u8; length];
    let mut read_len = 0usize;

    while read_len < length {
        let n = file.read(&mut data[read_len..])?;
        if n == 0 {
            break;
        }
        read_len += n;
    }

    if read_len > 0 {
        aspace.write(map_start, &data[..read_len])?;
    }
}
```

逐句理解：

#### 先判断是不是匿名映射

- 如果是匿名映射，就不需要文件内容；
- 如果不是匿名映射，就必须从 `fd` 对应的文件读取内容。

#### 用 `fd` 取得文件对象

`api::get_file_like(fd)` 的作用是：

- 从文件描述符表中找到对应文件；
- 得到一个可以执行 `read(...)` 的对象。

#### 分配一个内核缓冲区 `data`

```rust
let mut data = vec![0u8; length];
```

这不是映射区本身，而是一个“中转缓冲区”：

- 先把文件读到内核 buffer；
- 再从 buffer 写到用户地址空间。

#### 循环读文件

因为一次 `read` 不一定把想要的字节全部读满，所以使用循环：

- 每次尽可能继续往后读；
- 如果返回 0，说明到达文件末尾；
- 最终 `read_len` 记录实际读到了多少字节。

#### 把内容写入用户映射区

```rust
aspace.write(map_start, &data[..read_len])
```

这一步的效果是：

- 找到 `map_start` 对应的用户虚拟地址；
- 经过页表翻译，把数据写到它背后的物理页中；
- 于是用户程序随后访问 `addr` 时，看到的就是文件内容。

这就是文件映射在本题中的最核心实现思路。


### 8.9 第 9 步：返回起始地址

最后返回：

```rust
Ok(map_start.as_usize())
```

也就是把映射起始虚拟地址返回给用户程序。

用户程序拿到这个地址后，就可以把它当作普通指针一样读：

- `printf("%s", addr)`

在测试程序里，这样就会打印出：

- `Read back content: hello, arceos!`


## 9) 辅助函数的作用

### 9.1 `align_up_4k`

这个函数负责：

- 把字节数向上对齐到页大小；
- 防止整数溢出。

这是一个非常典型的内核辅助函数，因为：

- 用户接口按字节表达需求；
- 页表管理按页工作；
- 二者之间必须做一次转换。


### 9.2 `axerr_to_linux`

这个函数负责：

- 把 `axmm` / `axhal` 里的内部错误类型 `AxError`；
- 转换成 Linux 风格的 `LinuxError`；
- 这样系统调用就能以 errno 形式返回错误。

这一步体现了一个很重要的内核分层思想：

- 内部模块可以有自己的错误体系；
- 但系统调用面对用户时，必须转换成稳定的用户态 ABI 错误码。


## 10) 这份实现为什么能通过当前测试

因为当前测试程序只要求：

- 打开文件；
- 调 `mmap(NULL, 32, PROT_READ, MAP_PRIVATE, fd, 0)`；
- 然后读回字符串。

而本实现正好完成了这条链路：

1. 参数合法；
2. 内核找到空闲虚拟地址；
3. 建立映射并分配物理页；
4. 从文件读出 `hello, arceos!`；
5. 把内容写入映射页；
6. 返回映射地址；
7. 用户态读取成功。


## 11) 当前实现的边界

虽然已经通过练习，但要明确这还不是完整 Linux `mmap`：

### 11.1 `offset` 只做了合法性检查

当前实现：

- 检查了 `offset` 非负且页对齐；
- 但没有真正按 `offset` 调整文件读取位置。

也就是说：

- 目前默认总是“从文件当前位置/开头附近直接读”；
- 还没有实现真正的“从文件第 `offset` 字节处开始映射”。


### 11.2 `MAP_PRIVATE` / `MAP_SHARED` 只做了参数层区分

目前实现中：

- `MAP_PRIVATE` 和 `MAP_SHARED` 的差异主要体现在参数校验层面；
- 还没有实现真正的共享回写或写时复制（COW）语义。


### 11.3 `MAP_FIXED` 也只是最小支持

真正 Linux 下的 `MAP_FIXED` 语义比较强：

- 可能会覆盖原有映射；
- 对地址冲突处理有更复杂要求。

当前版本只是：

- 如果传了 `MAP_FIXED`，就尝试使用这个地址；
- 更完整的覆盖/替换逻辑并未展开实现。


### 11.4 还没有做 lazy file-backed mmap

当前是：

- 先把文件内容主动读入映射页。

更高级的做法可能是：

- 一开始只建立映射；
- 真正访问页面时再触发 page fault；
- page fault handler 按需从文件加载页内容。

但这显然已经超出本练习范围。

## 13) 这一题我真正学到的东西

这道题最大的收获不是“补了一个系统调用”，而是理解了下面这条链路：

1. 用户程序调用 `mmap`；
2. 系统调用进入内核；
3. 内核解析参数并在当前进程地址空间中选择虚拟地址；
4. 内核建立页表映射；
5. 如果是文件映射，再把文件内容灌入这段映射；
6. 用户程序随后把这段地址当普通内存访问。

也就是说，`mmap` 的本质是把“文件/内存资源”和“进程虚拟地址空间”接起来。

一旦理解了这一点，很多现象都会突然变得自然：

- 为什么 `mmap` 返回的是地址；
- 为什么要有 `prot` 和 `flags`；
- 为什么长度和偏移要按页对齐；
- 为什么需要地址空间管理器 `AddrSpace`；
- 为什么文件映射不只是“read 一下”这么简单。
